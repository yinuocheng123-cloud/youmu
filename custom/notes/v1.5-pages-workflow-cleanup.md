# 版本记录：v1.5-pages-workflow-cleanup

## 1. 本轮目标

本轮目标是在 V1.4 首页视觉体验与移动端重构完成并推送后，单独复核 GitHub Pages 部署链路，清理 `.github/workflows/static.yml` 与 `.github/workflows/pages.yml` 并存的问题。

本轮属于部署链路清理和发布范围收口，不属于首页结构、视觉、文案或功能调整。

核心目标：
- 最终只保留一个正式 GitHub Pages workflow。
- 正式 workflow 只发布 `_site`。
- `_site` 只包含公网运行必要文件。
- 避免 `custom/` 文档、截图、过程记录、甲方预览包、`.github/`、`.git/` 等内容发布到公网。

## 2. 背景与上下文

本轮承接：
- `custom/notes/v1.3-github-actions-pages-deploy.md`
- `custom/notes/v1.4-visual-mobile-experience.md`

V1.3 新增 `.github/workflows/pages.yml`，其设计目标是只把 `index.html`、`styles.css`、`script.js`、`assets/`、`data/`、`404.html`、`robots.txt`、`sitemap.xml`、`.nojekyll` 复制到 `_site` 后发布。

V1.4 记录中发现远端新增 `.github/workflows/static.yml`，该 workflow 与 `pages.yml` 并存，且使用 `actions/upload-pages-artifact` 上传 `path: '.'`。这意味着它可能把整个仓库作为 Pages artifact 发布。

## 3. 问题分析

本轮读取两个 workflow 后确认：

- `.github/workflows/pages.yml`
  - 触发条件：推送到 `master`，以及 `workflow_dispatch` 手动运行。
  - 权限：`contents: read`、`pages: write`、`id-token: write`。
  - 使用官方 Actions：`actions/checkout@v4`、`actions/configure-pages@v5`、`actions/upload-pages-artifact@v4`、`actions/deploy-pages@v4`。
  - 发布方式：先创建 `_site`，只复制公网运行必要文件，再上传 `_site`。
  - 风险：符合当前要求，风险较低。

- `.github/workflows/static.yml`
  - 触发条件：推送到 `master`，以及 `workflow_dispatch` 手动运行。
  - 权限：同样具备 Pages 部署权限。
  - 发布方式：`actions/upload-pages-artifact@v3`，`path: '.'`。
  - 风险：会上传整个仓库，可能包含 `custom/` 文档、截图、过程记录、甲方预览包、`.github/` workflow 文件和其他不应公开发布的文件。

核心问题不是页面代码，而是两个 workflow 同时具备部署权限，并且其中一个会把整个仓库作为 artifact。

## 4. 候选方案比较

### 方案一：保留两个 workflow，仅在文档里提醒

优点是改动最小。缺点是风险仍然存在，只要 `static.yml` 被 push 或手动触发，就可能发布整个仓库。

本轮放弃该方案。

### 方案二：禁用 `static.yml` 触发条件

做法是修改 `static.yml`，移除 `push` 和 `workflow_dispatch`。

优点是保留文件便于回溯。缺点是仓库仍存在一个废弃 workflow，后续维护者可能误启用，且与“最终只保留一个正式 workflow”的要求不完全一致。

本轮没有采用该方案。

### 方案三：删除 `static.yml`，只保留 `pages.yml`

优点是发布入口唯一，维护成本最低，风险最清晰。缺点是删除远端新增的示例 workflow 后，如果未来想参考 GitHub 默认模板，需要重新从文档查找。

本轮采用该方案。

## 5. 最终决策

最终保留 `.github/workflows/pages.yml` 作为唯一正式 GitHub Pages 部署 workflow。

删除 `.github/workflows/static.yml`。

正式发布目录为 `_site`。

正式发布内容范围仅包括：
- `index.html`
- `styles.css`
- `script.js`
- `assets/`
- `data/`
- `404.html`
- `robots.txt`
- `sitemap.xml`
- `.nojekyll`

明确不发布：
- `custom/`
- `.github/`
- `.git/`
- 截图
- 项目过程记录
- 甲方预览包
- 本地说明文档

## 6. 具体实现

本轮实际修改：
- 删除 `.github/workflows/static.yml`。
- 保留并复核 `.github/workflows/pages.yml`。
- 更新 `custom/github-pages-guide.md`，说明 V1.5 后正式部署入口只保留 `pages.yml`，并明确 `_site` 发布范围。
- 更新 `custom/launch-checklist.md`，增加 workflow 唯一性、`_site` 发布范围和排除 `custom/` 的检查项。
- 更新 `custom/client-preview-note.md`，补充 V1.5 部署链路清理说明。
- 新增本记录文件 `custom/notes/v1.5-pages-workflow-cleanup.md`。

本轮没有修改：
- `index.html`
- `styles.css`
- `script.js`
- `data/site-content.js`
- 图片、Logo、二维码或首页模块结构

## 7. 本轮优点

1. GitHub Pages 部署入口从两个收敛为一个，降低重复部署和互相覆盖风险。
2. 保留的 workflow 只发布 `_site`，避免把内部文档和过程资料公开。
3. 部署说明、上线清单和甲方预览说明保持一致，后续交接更清楚。
4. 没有触碰首页结构、视觉和文案，降低对 V1.4 视觉成果的回归风险。

## 8. 本轮缺点与代价

1. 删除 `static.yml` 后，GitHub Actions 页面历史里可能仍能看到此前运行记录，但仓库代码中不再保留该 workflow。
2. 本地只能验证 workflow 文件内容和脚本语法，无法完全替代 GitHub Actions 线上运行结果。
3. 本轮没有修改 `robots.txt`，当前仍是预览阶段不主动收录策略。

## 10. 本轮结论

V1.5 的核心结论是：GitHub Pages 正式部署链路应只保留 `.github/workflows/pages.yml`，并且只通过 `_site` 发布公网必要文件。

`custom/` 是项目交付文档、过程记录、截图和甲方预览包存放目录，不应进入 GitHub Pages artifact。
