# 版本记录：v1.4-visual-mobile-experience

## 1. 本轮目标

本轮目标是在 V1.3 已完成 GitHub Pages Actions 部署链路、且公网预览地址已经能够打开的基础上，单独完成“首页视觉体验与移动端重构版”。

本轮属于首页视觉体验优化、移动端可用性提升和交付说明补充，不属于后台、登录、商城、支付、数据库或真实厂商资料处理。

重点目标包括：
- 优化首页首屏视觉，让 Hero 标题、钩子文案和定位说明更清楚。
- 优化顶部 Logo 和导航，桌面端更克制，手机端不再用换行导航挤压页面。
- 强化好物方案分类，默认突出柚木整装和柚木地板。
- 强化企业微信二维码承接区，让用户更明确知道下一步如何添加顾问。
- 生成 V1.4 桌面端与手机端截图，更新甲方预览说明和预览包截图。

本轮承接以下记录：

- `custom/notes/v1.2-github-pages-preview-ready.md`：确认静态资源使用相对路径，适配 GitHub Pages 项目页。

本轮开始时执行了最新仓库同步。远端新增提交 `937eca7 Add GitHub Actions workflow for static site deployment`，新增 `.github/workflows/static.yml`。该文件与 V1.3 的 `.github/workflows/pages.yml` 并存。本轮不以部署链路为主，因此未重写 workflow，但后续建议单独复核两个 workflow 的发布范围是否一致。

## 3. 问题分析

修改前首页已经具备完整模块和内容，但存在三个体验问题：

1. Hero 使用整屏大图暗色遮罩，视觉气质足够，但主文案在手机端和较小屏幕上容易显得拥挤，第一眼不够明确。
2. 移动端顶部导航以换行方式展开，知识分类下拉会占据较多空间，不适合长页面浏览。
3. 好物方案虽然已经包含柚木整装、柚木地板、庭院户外、茶室会客和家具好物，但缺少更清楚的分类入口，用户第一眼容易把它理解成普通卡片列表。

核心矛盾是：既要让首页更成熟、更像可以继续给甲方预览的站点，又不能改变首页整体结构，更不能把网站做成家具商城。

## 4. 候选方案比较

### 方案一：重做首页结构

做法是重新设计首页模块顺序，拆分更多内容区甚至新增详情页。

优点是视觉自由度更高。缺点是会偏离本轮“不重做首页结构”的要求，也会增加真实资料和详情页承接压力。

本轮放弃该方案。

### 方案二：只改 CSS，不改 HTML

做法是在现有结构上仅通过样式覆盖优化视觉。

优点是改动最小。缺点是无法建立真正的移动端抽屉菜单，也无法让好物方案分类按钮具备清晰语义。

本轮没有采用纯 CSS 方案，而是采用“少量 HTML 增强 + CSS 覆盖 + JS 交互补充”。

### 方案三：保留首页模块顺序，增强首屏、导航、分类和承接区

做法是保留 Hero、三大价值、柚木知识、好物方案、推荐厂商、推荐标准、企业微信社群、Footer 的顺序，只在局部结构中增加必要语义和交互。

优点是符合本轮边界，能显著提升桌面端和移动端体验。缺点是旧 CSS 仍然保留，V1.4 采用文件末尾覆盖样式，后续如果继续大规模迭代，可能需要再做一次样式整理。

本轮采用该方案。

## 5. 最终决策

最终采用“保留首页模块顺序 + 局部结构增强 + V1.4 样式覆盖 + JS 交互补充”的方式。

关键决策：
- Header 使用喜字 mark 加“柚喜饰界”短字标，Footer 继续使用横版组合 Logo。
- Hero 改为左侧文案、右侧生活方式图，避免暗色背景压低文字可读性。
- 桌面端保留柚木知识和好物方案下拉，移动端改为菜单按钮和抽屉式导航。
- 好物方案增加分类标签，默认高亮柚木整装和柚木地板。
- 所有主要 CTA 继续滚动到页面内区域或企业微信承接区，不新增外部页面和商城路径。
- 企业微信区使用更深的木色背景，并增加手机端识别二维码提示。

## 7. 本轮优点

1. 首屏更清楚，用户第一眼能看见品牌名、核心钩子和网站定位，不再依赖暗色大图压字。
2. 手机端导航更像可用网站，不再把所有导航挤在顶部换行。
3. 好物方案从普通卡片列表升级为有分类意识的内容区，柚木整装和柚木地板重点更明确。
4. 企业微信承接区更像最终转化终点，二维码、备注和手机端识别提示更清楚。
5. 仍然保持低电商感，没有新增价格、购物车、立即购买、登录、支付或数据库入口。
6. 文档和截图同步更新，便于继续给甲方复核。

## 8. 本轮缺点与代价

1. 本轮采用 CSS 文件末尾覆盖方式，短期可控，但长期看样式文件会变厚。后续如果继续做 V1.5 或正式上线版，建议整理旧 Hero 和旧移动端导航样式，减少重复规则。
2. 移动端菜单已经可用，但当前仍是静态单页导航，没有加入更复杂的当前栏目高亮或滚动监听。
3. 好物方案分类标签当前采用滚动定位和高亮聚焦，不是完整筛选器；这样能保留所有内容可见，但如果后续内容增多，可能需要改为真正的分类筛选或详情页。
4. 真实厂商、真实案例、真实资质和最终二维码仍未处理，当前文案必须继续保持“样板位”“展示方向”等谨慎口径。
5. 远端新增的 `.github/workflows/static.yml` 与 V1.3 的 `.github/workflows/pages.yml` 并存，且 `static.yml` 当前从 `.` 上传 artifact。由于本轮不是部署 workflow 专项，没有直接修改它；但后续建议单独复核，避免公开发布 `custom/` 文档。

## 10. 本轮结论

V1.4 证明在不重做首页结构、不新增商城功能、不处理真实厂商背书的前提下，可以通过局部结构增强和样式覆盖显著提升首页的第一屏理解效率与移动端可用性。

本轮应继承的结论：
- 当前首页继续定位为“柚木知识平台＋好物方案＋推荐厂商展示＋企业微信社群承接入口”。
- 企业微信仍是当前核心承接方式。
- 柚木整装和柚木地板可以作为重点方向表达，但不能写成真实交付承诺。
- 推荐厂商区继续保持样板位和展示方向口径。
- 真实资料真实性、图片授权、二维码最终状态、联系方式和资质信息仍需正式上线前统一处理。

## 11. 下一轮建议

1. 单独复核 GitHub Actions workflow，尤其是远端新增 `.github/workflows/static.yml` 是否会把 `custom/` 文档发布到 Pages。
2. 如果继续做 V1.5，建议在不新增后台的前提下优化移动端菜单当前状态、方案分类筛选和首屏加载性能。
3. 正式上线前优先处理真实二维码、真实联系方式、真实授权图片、推荐厂商展示授权和备案信息。
