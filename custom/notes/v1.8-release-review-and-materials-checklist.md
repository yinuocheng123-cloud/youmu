# 版本记录：v1.8-release-review-and-materials-checklist

## 1. 本轮目标

本轮目标是在 `V1.7.7 推荐厂商改为厂商会员站体系` 的基础上，补齐正式发布前的人工作业文件，而不是继续扩栏目或重写页面。

本轮属于：

- 可维护性提升
- 发布前准备
- 风险控制

核心目标是建立一套可执行的上线前资料清单、人工验收清单、来源绑定清单、厂商会员站资料模板，以及一个能发现明显发布阻塞项的预检脚本。

## 2. 背景与上下文

当前站点已经完成以下基础：

- 五项一级导航结构稳定
- 内容页深度已恢复
- 乱码、链接、内容深度已有基础检查脚本
- “推荐厂商”页面内核已调整为“厂商会员站体系”

但正式发布前还缺少一类非常关键的产物：不是再写页面，而是明确告诉后续执行者“哪些真实资料还没补”“哪些地方不能假装已经完成”“哪些内容需要人工复核后才适合公开发布”。

## 3. 问题分析

表面问题不是页面缺功能，而是发布前信息状态不透明：

- 当前公开页面仍存在占位联系方式
- 当前二维码虽已接入图片，但是否为最终投放版本仍需人工确认
- 知识页已有通用来源方向，但没有逐页来源绑定清单
- 厂商会员站样板结构已经明确，但真实会员站资料收集模板还不完整
- 现有检查脚本能发现链接、乱码和内容深度问题，但不能直接提示“当前并不适合正式发布”的占位内容与风险表达

真正的核心矛盾是：

- 站点结构已经接近稳定
- 但上线所需的真实资料、授权确认和人工验收动作，还没有形成可交接的统一文档

## 4. 候选方案比较

### 方案 A：继续直接改页面内容

核心思路：

- 先把能看到的占位内容改掉
- 再边做边补资料

优点：

- 页面表面看起来更接近正式站

缺点：

- 没有真实资料时容易被迫编造
- 会让“预览站”和“正式站”边界再次模糊
- 不能沉淀可复用的上线流程

放弃原因：

- 与本轮“不要伪造真实资料、不要假装已经完成”的目标冲突

### 方案 B：建立发布前清单与预检体系

核心思路：

- 把资料缺口、人工验收项、来源绑定项和会员站收集模板先落成文档
- 新增一个正式发布预检脚本，专门发现明显上线阻塞项

优点：

- 不需要伪造真实资料
- 更符合当前站点仍是静态预览站的阶段定位
- 后续任何人接手都能知道从哪里开始补齐上线要件

缺点：

- 本轮不会直接把站点变成正式可发布状态
- 预检脚本会更清楚地暴露当前仍未补齐的占位项

选择原因：

- 最符合当前阶段目标，也最符合仓库“思路记录与可继承性优先”的规则

## 5. 最终决策

本轮最终采用方案 B。

关键设计决策：

- 不新增后台、登录、商城、支付、数据库
- 不改 `forms/form.js`
- 不改 GitHub Actions workflow
- 不改 sitemap
- 不扩一级导航
- 只在 `custom/` 下新增发布前文档、模板和脚本
- 说明文件只做增量补充，不做无关重写

## 6. 具体实现

本轮新增文件：

- `custom/release-materials-needed.md`
- `custom/public-review-checklist.md`
- `custom/source-binding-checklist.md`
- `custom/vendor-member-site-material-template.md`
- `custom/check-release-readiness.mjs`

本轮更新文件：

- `custom/client-preview-note.md`
- `custom/launch-checklist.md`
- `custom/content-source-log.md`

本轮各文件作用：

- `release-materials-needed.md`：集中列出正式发布前仍需补齐的真实资料
- `public-review-checklist.md`：作为人工验收勾选单
- `source-binding-checklist.md`：把通用来源池逐步落到重点页面
- `vendor-member-site-material-template.md`：作为未来收集真实厂商会员站资料的统一模板
- `check-release-readiness.mjs`：扫描公开页面与数据源，发现乱码、高风险词和明显占位项
- `client-preview-note.md`：补充 V1.8 发布前资料与验收说明
- `launch-checklist.md`：补充 V1.8 正式发布前专项复核段
- `content-source-log.md`：补充 V1.8 来源绑定说明

## 7. 本轮优点

相比之前，本轮的提升主要在于“发布前可执行性”：

- 可读性更高：上线前该补什么一目了然
- 可维护性更高：后续执行者不必重新猜测资料缺口
- 可扩展性更高：来源绑定和会员站资料收集有了统一模板
- 调试便利性更高：新增预检脚本后，能更早发现明显不适合上线的公开内容
- 风险控制更稳：强调没有真实资料时不伪造、不硬写

## 8. 本轮缺点与代价

这部分必须保留真实取舍：

- 本轮没有把站点直接推进到“完全可正式发布”
- 新增预检脚本后，当前站点会更清楚地暴露出占位联系方式与待确认资料
- 来源绑定清单目前仍以“待人工核验”为主，未形成逐页核验闭环
- 由于本轮不联网核验，也没有新增真实厂商和真实案例，所以很多条目只能停留在清单层

这是有意接受的代价，因为当前更重要的是先把“不能假装已完成”的边界写清楚。

## 9. 验证与结果

本轮运行了以下检查：

- `node --check script.js`
- `node --check data\site-content.js`
- `node --check forms\form.js`
- `node custom\check-home-links.mjs`
- `node custom\check-site-links.mjs`
- `node custom\check-encoding.mjs`
- `node custom\check-content-depth.mjs`
- `node custom\check-release-readiness.mjs`

验证结果说明：

- 既有语法、链接、编码和内容深度检查应继续通过
- `check-release-readiness.mjs` 用于发现当前正式发布阻塞项
- 如果脚本报出占位联系方式、备案待补或二维码待确认，这属于本轮预期结果，不应通过删免责声明来“伪通过”

## 10. 本轮结论

本轮最重要的结论是：

- 站点结构已经足够稳定，下一阶段的重点不再是扩页面，而是补真实资料和做人工作业闭环
- 当前项目最需要的不是“再写一版页面”，而是“建立正式发布前可执行的核查体系”
- 发布前资料清单、人工验收清单、来源绑定清单、厂商会员站资料模板和预检脚本，已经成为后续推进正式发布的基线资产

## 11. 下一轮建议

- 优先按 `custom/release-materials-needed.md` 补齐真实二维码、联系方式、备案信息和可公开资料
- 按 `custom/source-binding-checklist.md` 逐页完成重点知识页与文章页的来源人工核验
- 在有真实厂商授权材料后，按 `custom/vendor-member-site-material-template.md` 收集并替换真实会员站资料
