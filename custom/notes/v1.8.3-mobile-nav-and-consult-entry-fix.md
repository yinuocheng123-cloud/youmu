# 版本记录：v1.8.3-mobile-nav-and-consult-entry-fix

## 1. 本轮目标

本轮目标是单独完成“移动端导航全站可用与普通用户咨询入口下线”两件事。

优先级来自用户在 V1.8.2 后发现的两个体验问题：

- 普通用户没有必要再填写在线咨询页面，当前站点不是后台线索系统，也不是自动提交系统，应回到“社群交流 / 添加柚喜顾问”承接。
- 移动端导航在首页可用，但进入知识、方案、推荐厂商、关于等二级页面后无法下拉，影响全站基础浏览。

本轮目标属于：

- Bug 修复
- 用户路径修正
- 可维护性提升

## 2. 背景与上下文

本轮承接 `custom/notes/v1.8.2-secondary-category-index-pages.md`。V1.8.2 已经把二级栏目从说明页改成内容索引页，但二级页移动端导航缺少可用脚本和菜单结构，导致索引页在手机端反而不容易继续浏览。

同时，V1.8.1 与 V1.8.2 都确认当前项目仍是静态内容型预览站，不包含后台、登录、商城、支付或数据库。既然没有后台线索系统，普通用户继续被引导去填写在线咨询页会制造误解，因此本轮需要把前台承接口径改回社群和顾问沟通。

## 3. 问题分析

用户发现的问题有两个。

第一个问题是咨询入口不符合当前站点能力。原有 `forms/consult.html` 是静态摘要生成页，不会保存、不提交后台，也不形成自动工单。继续把它作为普通用户主入口，会让用户以为网站具备咨询表单处理能力。

第二个问题是移动端导航在二级页面失效。检查后判断，首页有完整的 `.site-header`、`data-menu-toggle`、`data-mobile-menu`、遮罩和关闭按钮结构，并加载根目录 `script.js`；但内容页多为 `.content-site-header` 与 `.content-nav`，缺少首页同款移动端菜单结构，且原先未全站加载 `../script.js`。因此问题不是单一按钮失效，而是“脚本路径 + Header 结构”两层不一致。

## 4. 候选方案比较

方案 A：逐页复制首页移动端菜单结构。

- 优点：HTML 结构直观，每页都有显式菜单。
- 缺点：需要大面积复制重复结构，后续导航改动容易漏改。
- 放弃原因：本轮只需要修复可用性，不适合把每个二级页面都变成首页 Header 的副本。

方案 B：内容页保留现有 Header，由全站脚本动态补齐移动端菜单。

- 优点：改动集中在 `script.js` 和少量 CSS，内容页只需正确加载脚本。
- 缺点：移动菜单结构依赖 JS 生成，如果脚本未加载仍不可用。
- 选择原因：最符合“最小改动”和“不大改视觉”的要求，也便于通过检查脚本约束二级页必须引用 `../script.js`。

方案 C：继续保留普通用户咨询表单作为备用入口，只改文案。

- 优点：改动小。
- 缺点：仍会强化“填写表单”的心智，与用户明确判断冲突。
- 放弃原因：用户已确认“填写咨询表单没有必要”，所以前台入口应下线，而不是换个说法继续保留。

## 5. 最终决策

本轮采用方案 B，并将普通用户咨询入口降级为备用说明页。

关键决策：

- 首页移动端菜单继续使用已有结构。
- 二级内容页继续保留 `.content-site-header` 和 `.content-nav`，不大改页面视觉。
- `script.js` 在发现内容页没有移动端菜单结构时，自动生成菜单按钮、菜单面板、遮罩、关闭按钮和“添加柚喜顾问” CTA。
- 根目录页面使用 `./script.js`，二级目录页面使用 `../script.js`。
- 普通用户 CTA 不再指向 `forms/consult.html`，统一指向首页企业微信区。
- `forms/consult.html` 保留为“联系柚喜顾问”备用说明页，不再加载 `forms/form.js`。
- `forms/vendor-apply.html` 继续保留，并继续加载 `forms/form.js` 处理会员站申请摘要。

## 6. 具体实现

本轮涉及文件包括：

- `script.js`：改为全站通用移动菜单逻辑，支持内容页动态补齐菜单结构。
- `styles.css`：补充内容页移动端 Header 与动态菜单按钮样式。
- 全站 HTML 页面：补充根目录 / 二级目录脚本引用，并清理普通用户咨询入口。
- `forms/consult.html`：改为备用联系说明页。
- `forms/vendor-apply.html`：保留会员站申请表，同时补充 `../script.js` 以支持移动端导航。
- `custom/check-release-readiness.mjs`：补充二级页面脚本 / 样式路径检查与普通用户咨询入口文案检查。
- `custom/client-preview-note.md`：补充 V1.8.3 预览说明。
- `custom/launch-checklist.md`：补充 V1.8.3 专项复核清单。

移动端导航修复情况：

- 根目录页面加载 `./script.js`。
- 二级页面加载 `../script.js`。
- 内容页移动端菜单 CTA 指向 `../index.html#wechat`。
- 点击菜单按钮可打开，点击关闭按钮、遮罩、菜单项或按 Escape 可关闭。

咨询入口下线情况：

- 前台不再主动链接 `forms/consult.html`。
- “填写咨询摘要”“咨询摘要表”等普通用户表单心智文案已调整为“加入社群交流”“添加柚喜顾问”或“沟通问题”。
- Footer 沟通入口改为社群 / 顾问承接。

## 7. 本轮优点

- 移动端导航修复集中，不需要逐页复制首页菜单结构。
- 二级页面仍保持 V1.8.2 的索引化结构，不重新扩栏目。
- 普通用户承接方式更符合当前静态预览站能力，避免暗示后台收集线索。
- `forms/consult.html` 未删除，保留了历史路径的温和说明，减少旧链接访问时的断层。
- `check-release-readiness.mjs` 能提前发现二级脚本路径错误和咨询入口回流问题。

## 8. 本轮缺点与代价

当前方案仍有代价：

- 内容页移动菜单结构由 JS 动态生成，浏览器禁用 JS 时移动菜单不可用。
- 本轮没有做真实手机设备验收，只能通过本地浏览器和脚本检查初步确认。
- `forms/consult.html` 虽然保留，但已不再承担表单功能；如果未来真的需要后台线索系统，需要重新设计而不是恢复旧静态页。
- 全站 HTML 都增加了脚本引用，提交 diff 涉及页面较多，但每个页面的业务结构没有重写。

## 10. 本轮结论

本轮最重要的结论是：当前阶段普通用户承接应优先回到社群和顾问沟通，而不是静态咨询表单。移动端导航则不应依赖首页专属结构，必须让内容页也能加载同一套菜单脚本能力。

本轮确认：

- 没有新增后台、登录、商城、支付、数据库。
- 没有修改 GitHub Actions workflow。
- 没有修改 sitemap。
- 没有修改 `forms/form.js` 的核心摘要生成逻辑。
- 没有新增真实厂商、真实案例、真实认证或真实资质。
