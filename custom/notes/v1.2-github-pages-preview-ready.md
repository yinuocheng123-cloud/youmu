# 版本记录：v1.2-github-pages-preview-ready

## 1. 本轮目标

本轮目标是在 V1.1 已完成首页交互闭环与内容补齐的基础上，单独完成“GitHub Pages 公网预览与上线前适配版”。

本轮属于 GitHub Pages 预览适配、基础 SEO 补齐、静态站点发布前检查和可维护性提升。优先级是让仓库根目录可以作为 GitHub Pages 静态站发布，并让甲方能够通过公网地址确认首页效果。

本轮不新增后台、登录、商城、支付、数据库，不把网站改成商城，不新增购物车、价格或“立即购买”，不编造真实厂商、真实案例、真实认证或真实资质。

## 2. 背景与上下文

本轮承接 `custom/notes/v1.1-home-interaction-and-content-completion.md`。V1.1 已经确认首页保持单页结构，所有主要导航、按钮和 CTA 已形成页面内或企业微信承接闭环。

同时继续继承 V0.9 与 V1.0 的素材边界：当前图片为本地开放授权图库素材，用于 demo 和甲方预览，不代表真实厂商案例、真实产品资料或最终上线素材。

当前远端仓库为 `https://github.com/yinuocheng123-cloud/youmu.git`，本轮预期 GitHub Pages 地址暂按 `https://yinuocheng123-cloud.github.io/youmu/` 准备。

## 3. 问题分析

当前首页主体已经稳定，核心问题不是页面结构，而是上线前静态站点发布所需的细节尚未完全补齐：

- GitHub Pages 项目页带有 `/youmu/` 子路径，资源不能使用 `/assets/...` 这类根路径。
- 公网预览阶段需要 `.nojekyll`、`404.html`、`robots.txt`、`sitemap.xml` 等基础文件。
- SEO 和社交分享元信息需要包含明确的 `og:url` 与 `og:image`。
- 甲方预览说明需要明确：公网预览不等于正式上线，真实资料和授权仍需上线前确认。
- 上线检查清单需要补充 GitHub Pages 专项检查项。

核心矛盾是：要让页面具备公网预览条件，但不能借此把 demo 内容包装成正式上线资料，也不能扩大功能范围。

## 4. 候选方案比较

### 方案一：只新增 Pages 文件，不改首页元信息

优点是改动小，能快速满足部分 GitHub Pages 文件需求。

缺点是 Open Graph 分享时缺少明确页面地址和图片地址，SEO 检查不完整；甲方说明和上线清单也无法承接本轮状态。

结论：放弃作为完整方案，仅保留“新增 Pages 文件”的部分。

### 方案二：补齐基础 Pages 文件、SEO 元信息和说明文档

优点是改动范围仍然很小，但能覆盖公网预览、路径检查、基础 SEO、甲方说明和后续交接记录。

缺点是仍不能自动开启 GitHub Pages；如果仓库设置中未启用 Pages，仍需人工到 GitHub Settings 中开启。

结论：采用。

### 方案三：直接配置完整正式上线 SEO 和允许搜索引擎收录

优点是更接近正式发布。

缺点是当前资料、图片、真实厂商、二维码、联系方式和备案信息仍未最终确认，提前允许收录风险较高。

结论：放弃。本轮按“公网预览，不主动收录”处理。

## 5. 最终决策

最终采用“GitHub Pages 基础文件补齐 + 首页 head 元信息小幅优化 + 文档和记录同步”的方案。

关键决策：

- 新增 `.nojekyll`，避免 GitHub Pages 对静态资源做 Jekyll 处理。
- 新增 `404.html`，提供与当前品牌风格一致的简单错误页，并用 `./index.html` 返回首页。
- 新增 `robots.txt`，当前设置为 `Disallow: /`，表示公网预览阶段不主动让搜索引擎收录。
- 新增 `sitemap.xml`，先保留 GitHub Pages 预览地址占位，正式上线后再按最终域名更新。
- 新增 `custom/github-pages-guide.md`，给非技术人员说明如何开启 GitHub Pages。
- 更新 `index.html` 和 `data/site-content.js` 中的 SEO/OG 信息，保持克制，不写夸大承诺。

## 7. 本轮优点

- 仓库根目录已经具备 GitHub Pages 静态发布所需的基础文件。
- 首页资源路径继续保持相对路径，适合 `/youmu/` 子路径访问。
- SEO 和社交分享元信息更完整，同时保持“预览版”阶段的克制表达。
- `robots.txt` 明确当前不主动收录，降低 demo 内容被提前索引的风险。
- 甲方预览说明、上线清单和技术开启说明都已同步，后续交接成本更低。
- 本轮没有扩大功能范围，没有新增电商、后台或数据库能力。

## 8. 本轮缺点与代价

当前方案仍不能自动保证 GitHub Pages 已在 GitHub 仓库设置中开启。如果 Pages 未启用，需要人工进入 GitHub `Settings -> Pages` 设置 `master` 分支和 `root` 目录。

`robots.txt` 当前禁止抓取，适合公网预览阶段，但正式上线允许收录时必须调整，否则搜索引擎仍不会主动收录页面。

`sitemap.xml` 当前只有首页地址，适合当前单页静态站；后续如果新增知识详情页、方案页或厂商页，需要同步扩展 sitemap。

图片没有在本轮压缩。当前最大图片约 654KB，预览阶段可接受，但正式上线前如果替换真实图片，应重新做统一压缩与加载速度复核。

## 10. 本轮结论

V1.2 的核心结论是：在不改动首页结构、不新增功能、不引入远程资源的前提下，当前项目已经可以按 GitHub Pages 根目录静态站进行公网预览准备。

本轮继续继承 V1.1 的首页交互闭环、V1.0 的知识与方案结构、V0.9 的本地开放授权素材策略，并新增了 GitHub Pages 预览阶段的发布说明和基础文件。

资料真实性仍最后统一处理。当前页面仍是预览版，不代表正式上线内容、真实厂商背书或真实交付承诺。

2. GitHub Pages 可访问后，打开 `https://yinuocheng123-cloud.github.io/youmu/` 做公网端图片、Logo、favicon、CTA 和移动端复核。
