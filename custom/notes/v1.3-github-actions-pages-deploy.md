# 版本记录：v1.3-github-actions-pages-deploy

## 1. 本轮目标

本轮目标是在 V1.2 已完成 GitHub Pages 静态资源适配、但公网地址 `https://yinuocheng123-cloud.github.io/youmu/` 仍返回 404 的基础上，新增 GitHub Actions Pages 部署 workflow。

本轮属于部署流程补充和上线前发布链路修复，不属于首页结构调整或功能开发。

明确边界：

- 不重做首页结构。
- 不新增后台、登录、商城、支付、数据库。
- 不增加购物车、价格、“立即购买”。
- 不编造真实厂商、真实案例、真实认证或真实资质。
- 不引入远程图片、远程字体、远程脚本。

## 2. 背景与上下文

本轮承接 `custom/notes/v1.2-github-pages-preview-ready.md`。

V1.2 已经完成：

- 根目录静态站点基础文件：`.nojekyll`、`404.html`、`robots.txt`、`sitemap.xml`。
- GitHub Pages 项目页相对路径检查。
- SEO 与 Open Graph 基础信息。
- `custom/github-pages-guide.md` 中的 Pages 手动开启说明。

但复查后发现：

- GitHub 源码直链 `raw.githubusercontent.com/yinuocheng123-cloud/youmu/master/index.html` 可访问，说明代码已在远程仓库。
- `https://yinuocheng123-cloud.github.io/youmu/` 与 `https://yinuocheng123-cloud.github.io/youmu/index.html` 仍返回 GitHub Pages 404。
- 这说明问题更可能集中在 Pages 发布设置、发布源选择或 GitHub Pages 构建发布流程，而不是首页代码本身。

## 3. 问题分析

当前 404 的表面现象是公网地址没有产出站点。

核心问题是：分支发布方式可能没有真正生效，或者仓库 Settings 中 Pages 发布源没有正确生成部署。继续只检查 `master` 和 `/ (root)` 可能会卡在同一条路径上。

本轮判断：

- 静态站点本身已经满足根目录发布条件。
- 不需要改首页结构，也不需要改资源路径。
- 更合理的补充是新增 GitHub Actions Pages workflow，让 Actions 明确构建 Pages artifact 并调用官方部署 Action。

## 4. 候选方案比较

### 方案一：继续只使用分支发布

做法是继续要求在 Settings 中选择 `Deploy from a branch`、`master`、`/ (root)`。

优点是配置简单。缺点是当前已经出现保存后仍 404 的情况，继续重复同一路径不一定能解决问题。

本轮不再只依赖该方案。

### 方案二：新增 GitHub Actions Pages 部署

做法是新增 `.github/workflows/pages.yml`，在推送到 `master` 时自动把静态站点文件复制到 `_site`，再通过 `actions/upload-pages-artifact` 和 `actions/deploy-pages` 发布。

优点是发布链路更明确，可以在 GitHub `Actions` 页面看到每一步是否成功，也能绕过分支发布入口可能卡住的问题。缺点是需要仓库 Pages Source 切换或确认为 `GitHub Actions`，并且必须推送后由 GitHub 运行 workflow 才能看到结果。

本轮采用该方案。

### 方案三：改用第三方部署平台

例如 Vercel、Netlify 或其他静态站平台。

优点是部署反馈通常更直观。缺点是当前任务目标明确是 GitHub Pages，不宜引入新的平台、账号或部署依赖。

本轮放弃该方案。

## 5. 最终决策

最终采用“GitHub Actions Pages 部署 workflow + 文档同步说明”的方案。

关键决策：

1. workflow 只打包公网运行必需文件，不把 `custom/` 交付文档和截图发布到 Pages artifact。
2. 保留 V1.2 的分支发布说明作为备选，但新增 V1.3 Actions 发布优先说明。
3. workflow 使用 GitHub Pages 官方 Action：
   - `actions/configure-pages@v5`
   - `actions/upload-pages-artifact@v4`
   - `actions/deploy-pages@v4`
4. 触发方式包括推送 `master` 和手动 `workflow_dispatch`。

## 6. 具体实现

本轮新增：

- `.github/workflows/pages.yml`

workflow 内容包括：

- `on.push.branches: master`
- `workflow_dispatch`
- `permissions.contents: read`
- `permissions.pages: write`
- `permissions.id-token: write`
- `concurrency.group: pages`
- 构建目录 `_site`
- 复制运行必需文件：
  - `index.html`
  - `styles.css`
  - `script.js`
  - `404.html`
  - `robots.txt`
  - `sitemap.xml`
  - `.nojekyll`
  - `assets/`
  - `data/`
- 上传 `_site` 作为 Pages artifact。
- 调用 `actions/deploy-pages@v4` 部署。

本轮提交前复查时，将 `actions/upload-pages-artifact` 从 `@v3` 升级为 `@v4`。原因是用户要求优先参考当前 GitHub 官方示例中可用版本，且 `@v4` 与 `actions/configure-pages@v5`、`actions/deploy-pages@v4` 组合更符合当前 Pages custom workflow 示例。

同时新增 `.gitignore`，加入 `_site/`，避免本地临时打包目录被误提交。

本轮同步更新：

- `custom/github-pages-guide.md`：补充 V1.3 Actions 发布说明。
- `custom/launch-checklist.md`：补充 Actions workflow、Source 为 GitHub Actions、Actions 运行成功等检查项。
- `custom/client-preview-note.md`：补充 V1.3 GitHub Actions Pages 部署说明。
- `custom/client-review-package/README.md`：补充分支发布仍 404 时查看 Actions workflow。
- `custom/client-review-package/homepage-preview-note.md`：补充 V1.3 workflow 说明。

## 7. 本轮优点

1. 不改首页结构，只补部署链路，改动范围清晰。
2. GitHub Actions 页面可以看到部署过程，有利于定位构建失败、权限不足或 Pages Source 配置问题。
3. `_site` 只包含公网运行必需文件，避免把交付文档、截图、笔记等内部资料发布到公网。
4. 继续沿用 V1.2 的相对路径和 GitHub Pages 项目页地址策略。

## 8. 本轮缺点与代价

1. workflow 必须推送到 GitHub 后才会真正运行，本地无法完整验证 GitHub Pages 部署结果。
2. 如果仓库 `Settings -> Pages` 仍未切换到 `GitHub Actions`，或者仓库权限不允许 Actions 部署 Pages，公网地址可能仍然 404。
3. 当前 `robots.txt` 仍为 `Disallow: /`，适合预览阶段；正式上线前如果要收录，需要调整。
4. Actions 部署后仍需要到公网地址复核图片、Logo、favicon、CTA 和移动端显示。

## 9. 验证与结果

本轮本地可验证内容：

- `node --check script.js`
- `node --check data\site-content.js`
- `node custom\check-home-links.mjs`
- 资源路径检查
- 禁用商城/夸大表达检查
- workflow 文件内容检查

本轮无法在本地完成的内容：

- GitHub Actions workflow 的真实运行结果。
- GitHub Pages artifact 的线上部署结果。

原因是 GitHub Pages 部署必须在 GitHub 仓库推送后，由 GitHub Actions 环境执行。

## 10. 本轮结论

V1.3 的核心结论是：当前 404 不再优先按首页代码问题处理，而应通过 GitHub Actions Pages workflow 建立更可观察的部署链路。

后续默认继承：

- 首页继续保持静态站结构。
- 公网发布优先检查 Actions 的 `Deploy static site to GitHub Pages` 运行状态。
- 如果 Actions 成功但地址仍 404，再检查仓库 Pages Source、仓库可见性、Actions 权限和 Pages 环境状态。

## 11. 下一轮建议

1. 推送本轮 workflow 后，在 GitHub `Actions` 页面查看 `Deploy static site to GitHub Pages` 是否运行成功。
2. 在 `Settings -> Pages` 中确认 Source 为 `GitHub Actions`。
3. Actions 成功后访问 `https://yinuocheng123-cloud.github.io/youmu/`，并复核桌面端、移动端、图片、Logo、favicon 和 CTA。
