# 版本记录：v1.15.3a-frontstage-logic-language-cleanup

## 1. 本轮目标

本轮目标是单独完成“V1.15.3A 前台网站逻辑说明与内部话清理”。重点不是重做网站结构，也不是继续扩展页面数量，而是把公开前台页面里残留的“解释网站如何设计、栏目如何承接、资料如何核对”的表达，调整为用户能直接理解的内容价值、生活场景和下一步选择。

本轮属于公网验收后的前台表达收口。它承接 V1.15.2 的版式修补，但不改导航结构、不改栏目路径、不改正文大结构、不新增后台、登录、商城、支付或数据库。

## 2. 背景与上下文

当前基础版本为 `7a9a352 V1.15.2 栏目名统一为柚木好物`。此前 V1.15.1 已完成全站导航下拉结构统一，V1.15.2 已处理首页首屏、CTA、知识区强提示和重点内页版式细节。

上一轮 V1.15.3 大任务曾因网络中断停止，状态检查确认没有留下本地改动。因此本轮缩小范围，只处理第一阶段：公开前台表达去说明书化。

本轮开始前尝试执行 `git pull --ff-only`，但 GitHub 返回 403，当前环境无法拉取远端。由于本地 `master` 显示与 `origin/master` 同步且工作区干净，本轮在本地现有状态上继续推进，并在最终结果中保留该风险说明。

## 3. 问题分析

当前公开页面的主要问题不是内容缺失，而是部分段落仍在向用户解释“网站怎么组织内容”。例如知识页底部的“从知识继续往哪里走”“企业资料如何整理和核对”，会让页面像内部运营说明；首页推荐厂商区的“资料说明”“资料确认方向”“资料展示框架”，会让用户感到页面仍处在样板阶段；关于页和推荐厂商页中的“路径、承接、核对顺序”等表达，也会把网站逻辑暴露在前台。

这些表达放在开发记录中是有价值的，但放在公开页面会削弱正式感。前台应该让用户看到的是：柚木有什么价值、适合哪些生活场景、下一步可以看什么或去哪里交流。

## 4. 候选方案比较

方案一是继续执行完整 V1.15.3 大任务，全面清理所有页面和视觉细节。优点是覆盖彻底，缺点是范围过大，容易重复中断前的大任务，也可能超出用户本轮要求。

方案二是只修知识页截图问题。优点是速度快，缺点是首页、关于页、推荐厂商和观察库仍会保留同类说明书表达，后续还会反复返工。

方案三是按本轮边界做小范围全站同类收口：不改结构，只定位公开页面和数据源中的说明书式表达并改成自然前台文案，同时升级预检脚本。最终选择方案三，因为它覆盖同类问题，同时仍保持最小改动。

## 5. 最终决策

本轮采用“局部文案替换 + 数据源同步 + 预检规则升级”的方式。公开 HTML 中只处理确认为内部话、模板味或旧栏目名的表达；`data/site-content.js` 同步改掉旧前台栏目名“好物方案”和资料框架表述，避免后续内容源恢复旧文案；检查脚本新增本轮禁用词，保证后续不会重新引入。

厂商申请页保留“生成申请摘要”功能，因为这是现有静态表单的核心功能，不属于普通用户咨询表单入口，也没有修改 `forms/form.js`。

## 6. 具体实现

- `knowledge/index.html`：将“从知识继续往哪里走”改为“继续探索柚木生活”，把栏目动线说明改成柚木好物、推荐厂商和社群交流三个自然选择。
- `index.html`：清理首页 Hero、价值卡片、推荐厂商区、社群区和 Footer 中的“资料展示框架”“资料说明”“资料确认方向”“案例展示框架”等表达。
- `about/index.html`：保留 V1.15.2 的标题微调，把“路径、承接、核对”的内部说明改成用户看材料、看空间、看企业资料的自然表达。
- `solutions/index.html`：把“企业资料页会如何组织资料”的说明改成查看产品方向、授权图片、服务区域和服务边界。
- `vendors/index.html`：把推荐厂商页的“资料阅读秩序”和“核对顺序”改成用户查看企业资料页、企业准备清晰资料的表达。
- `vendors/candidates/index.html`：收紧“资料如何说明、如何组织、核对授权”等表述，改成关注产品资料、材料特征和服务边界。
- `knowledge/teak-*.html` 与 `articles/index.html`：去掉普通咨询表单残留说明，回到社群交流和人工资料确认。
- `data/site-content.js`：同步“柚木好物”栏目名，清理“好物方案”“资料展示框架”“案例展示框架”等旧数据源表达。
- `custom/check-release-readiness.mjs`：新增本轮说明书式表达和旧栏目名拦截，并增加 `data/site-content.js` 数据源检查。
- `custom/check-content-depth.mjs`：将推荐厂商导航检查改为只接受“推荐厂商”，不再把“会员站”作为前台导航可接受名称。
- `custom/client-preview-note.md`、`custom/launch-checklist.md`、`custom/public-review-checklist.md`、`custom/notes/v1.15.2-homepage-and-inner-layout-polish.md`：补充 V1.15.3A 的验收说明和承接记录。

## 7. 本轮优点

本轮没有改变页面结构和业务定位，却消除了用户最容易感知到的“说明书味”。知识页底部变成自然的继续探索入口，首页与推荐厂商区更像正式前台内容，公开资料观察库也更清楚地保持观察属性。

预检脚本升级后，旧栏目名“好物方案”和本轮重点内部话会被自动拦截，降低后续返工概率。

## 8. 本轮缺点与代价

本轮是表达收口，不是全站内容重写。部分长尾文章、案例和样板页仍可能保留“资料确认、人工核验”等必要边界说明，只要不构成模板味或内部运营话，本轮没有强行改写。

另一个代价是 `check-release-readiness.mjs` 中的禁用词更严格，未来如果确实需要在公开页面解释某个边界，需要优先用自然用户语言，而不是绕过检查。

## 10. 本轮结论

本轮确认：公网验收阶段最需要保护的是用户读到页面后的第一理解。网站逻辑可以存在于结构和检查脚本里，但不应该大面积出现在前台文案里。前台应优先讲内容价值、生活场景、资料边界和下一步选择。
