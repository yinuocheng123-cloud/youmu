# 版本记录：v1.15.3g-single-line-headings-and-layout-alignment-polish

## 1. 本轮目标

本轮目标是完成“V1.15.3G 标题单行与版式对齐收口”。它不是一次内容扩展，也不是栏目结构重做，而是针对公网前台已经暴露出的细节问题做小而完整的版式修订：短标题尽量稳定单行，避免一两个字孤立掉行；按钮组、入口链接和短标题卡片保持底线与视觉节奏一致；提示块和说明块不再抢占阅读中心。

本轮属于前台版式与文案微调，不新增后台、登录、商城、支付、数据库，不修改 GitHub Actions workflow，不修改 sitemap，不修改 `forms/form.js` 核心摘要逻辑，也不改变栏目路径结构。

## 2. 背景与上下文

本轮承接 `76f28d3 V1.15.3F-3 知识文章自然长文体收口`。V1.15.3F-3 已经把知识文章继续推向自然长文体，减少目录式、说明书式和碎片跳转式表达。当前问题转向视觉细节：页面内容方向已经基本稳定，但部分短标题、按钮和模块入口在桌面端仍会出现换行不自然、底线不齐或像补丁一样卡在页面中的情况。

V1.15.3D 曾经采用“少量 HTML 修正 + CSS 统一覆盖层”的方式做过视觉和移动端收口，本轮延续这个思路，不重新设计页面，只在现有结构上补一层更聚焦的短标题、CTA、模块头部和卡片标题规则。

## 3. 问题分析

本轮主要问题不是单个页面的一个错字，而是同类版式问题在多个页面中重复出现。短标题如果被容器宽度、字距或按钮挤压影响，桌面端容易出现“最后一个字掉到下一行”的问题；按钮组如果各自高度、内边距和基线不一致，就会显得像临时拼接；模块标题区如果同时塞入强 CTA 或过长入口文案，会破坏用户先读内容、再进入下一步的节奏。

核心矛盾是：前台页面需要看起来像正式网站，而不是开发迭代过程中的注释版、提示版或测试版。解决方式不能继续在每个页面散乱打补丁，否则样式会越来越碎；更稳妥的做法是在 `styles.css` 中补充统一规则，再对确实过长的短文案做少量前台化压缩。

## 4. 候选方案比较

方案一是逐页手工调整每一个标题和按钮。优点是局部可控，缺点是容易漏掉同类问题，也会让 HTML 结构越来越散，因此没有采用。

方案二是只改 CSS，完全不碰 HTML 文案。优点是改动集中，缺点是部分入口文案本身过长，强行单行会造成按钮过宽或移动端挤压，因此只作为本轮的一部分采用。

方案三是“统一 CSS 收口 + 少量短文案压缩”。CSS 负责短标题单行保护、CTA 基线和模块头部节奏；HTML 只处理确实容易导致按钮掉行或标题孤行的短文案。这个方案改动范围小，风险低，也符合本轮不改业务结构的边界，因此最终采用。

## 5. 最终决策

本轮采用方案三。短标题通过 `.nowrap-heading` 和模块标题统一规则保护桌面端单行，移动端恢复自然换行并通过 `word-break` 和 `text-wrap` 降低孤字概率。CTA 统一使用 `inline-flex` 的对齐方式和最低高度，保证同组按钮同高同底线。模块头部统一标题、说明和入口链接的垂直关系，让首页和内页的栏目节奏更一致。

文案层面只压缩少量会造成按钮过宽或短标题断裂的前台文案，例如把“申请建立推荐厂商资料页”压缩为“申请资料页”或“申请推荐厂商资料页”，把首页价值卡标题压缩为更短、更适合卡片阅读的说法。

## 6. 具体实现

- `styles.css`：新增 V1.15.3G 收口规则，覆盖短标题单行、页面 Hero 标题平衡、CTA 同高同底线、模块头部对齐、卡片标题断行保护和移动端标题换行策略。
- `about/index.html`：为“柚喜饰界的使命”“柚喜饰界带你怎么看？”补充短标题单行保护，与已有“为什么从柚木开始”“柚木生活场景”保持一致。
- `index.html`：压缩首页价值卡标题、推荐厂商入口按钮和家具方向卡片标题，减少桌面端按钮掉行和卡片标题断裂风险。
- `vendors/index.html`：压缩推荐厂商页入口按钮和申请区标题，让按钮组与模块标题更稳定。
- `forms/vendor-apply.html`：压缩申请页标题、面包屑、表单按钮和申请对象卡片标题，避免短标题在卡片内被硬拆成两行。
- `custom/notes/v1.15.3g-single-line-headings-and-layout-alignment-polish.md`：新增本轮版本记录，沉淀处理范围、取舍和验证结果。

## 7. 本轮优点

本轮没有重新做页面结构，而是把同类问题集中到统一样式规则中处理。这样可以让首页、关于页、推荐厂商页和申请页的短标题、CTA 和模块头部获得一致节奏，也减少后续继续点状修补的概率。

文案压缩只发生在前台短标题、按钮和卡片标题上，没有改变业务含义。用户看到的表达更利落，页面也更像正式网站，而不是带着迭代痕迹的说明稿。

## 8. 本轮缺点与代价

当前 `styles.css` 已经存在多轮版本覆盖层，本轮继续在末尾新增收口规则，短期风险较低，但长期不是最干净的样式组织方式。未来如果继续做大规模视觉升级，建议单独整理 Hero、模块头部、按钮、卡片和表单的设计系统，把历史覆盖层合并。

另一个代价是部分长按钮被压缩为更短表达，例如“申请资料页”。它更适合当前版式，但信息量比原文略少，因此需要依靠上下文标题和正文说明补足含义。

本轮还通过本地预览抽查首页、关于页、柚木知识首页、柚木好物首页、推荐厂商页、公开资料观察库页和厂商申请页。桌面端重点检查短标题单行、CTA 底线和说明块位置；移动端重点检查横向溢出和短标题孤字断裂。

验证结果：上述脚本检查全部通过。本地预览抽查中，7 个核心页面桌面端均无页面级横向溢出，首页 Hero 三枚 CTA 同顶线、同底线、同高度，申请页表单按钮同顶线、同底线、同高度；桌面端短标题和按钮文字未出现真实换行。390px 移动端抽查中，7 个核心页面均无横向溢出，也未发现短标题异常多行断裂。

## 10. 本轮结论

V1.15.3G 的结论是：当前阶段最值得做的是把已有前台内容磨到“正式网站”的状态，而不是继续增加说明和入口。短标题、按钮、模块头部和提示块这类细节会显著影响公网第一眼感受，应优先用统一样式和少量自然文案压缩处理。

本轮没有新增后台、登录、商城、支付、数据库，没有修改 workflow、sitemap、导航结构、栏目路径或 `forms/form.js` 核心逻辑。公网预览地址仍为：https://yinuocheng123-cloud.github.io/youmu/

资料真实性处理说明：本轮只处理版式和前台短文案，不新增真实企业、真实案例、授权图片、平台认证或交易担保。推荐厂商资料、公开资料观察库和申请页仍需人工确认资料来源、图片授权、联系方式、服务区域和服务边界。

## 11. 下一轮建议

1. GitHub Pages 部署后，优先用桌面端和手机端复查首页、关于页、推荐厂商页和申请页，确认短标题没有再次出现孤字掉行。
2. 如果后续继续做视觉升级，建议单独安排一次 `styles.css` 历史覆盖层整理，把 V1.15.2 到 V1.15.3G 的收口规则合并成更稳定的设计系统。
3. 内容层面不要再新增“如何使用本站”“继续看”“领取资料”这类提示块，除非它们被放在非常明确且不打断主阅读的位置。
