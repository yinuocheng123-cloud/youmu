# 版本记录：v1.20.5-fix-good-things-top-category-card-residue

## 1. 本轮目标

本轮目标是继续修复“柚木好物”页顶部分类卡残留问题：顶部六类按钮下方不得再出现六张大分类卡、分类说明文案或“更多内容”，同时把这些说明内容分别移入对应栏目 panel 的末尾。

本轮属于 Bug 修复与发布前体验收口。用户截图显示公网或页面视觉上仍能看到六张大分类卡，因此需要在源码、样式和检查脚本三层同时确认。

## 2. 背景与上下文

本轮承接 `v1.20.4-good-things-category-tabs-merge.md`。上一轮已经删除顶部 `good-category-card` 结构，并推送 `aab3ed9`。但旧卡片说明文案没有以栏目内部收尾说明块的方式落位，检查脚本也主要检查第一个 panel 之前的整体区域，未精确锁定“六类按钮结束后到第一个栏目 panel 之前”的视觉空隙。

本轮不修改 Header、sitemap、workflow、forms/form.js，不修改六类锚点名称，不修改 30 个好物文章 URL 和正文，也不修改知识文章正文。

## 3. 问题分析

问题表面是顶部六类按钮下方仍被反馈存在六张大卡。源码复核时，当前 `solutions/index.html` 顶部已经没有 `good-category-card`，但旧说明文案也没有作为栏目内收尾块明确出现，容易造成“删除了入口卡，但没有完成迁移”的交付缺口。

核心矛盾是顶部入口只能承担选择功能，分类说明必须属于对应栏目内容。检查脚本也应检查真正的顶部视觉空隙：从 `good-category-tabs` 的 `</nav>` 之后，到第一个 `good-category-panel` 之前。

## 4. 候选方案比较

方案一：只声明公网缓存，代码不改。

优点是改动最小；缺点是没有完成用户要求的“把卡片内容挪入栏目内部最后”，也不能进一步防止类似残留，因此放弃。

方案二：把旧卡片原样移入栏目内部。

优点是迁移直观；缺点是继续使用 `good-category-card` / `good-category-more` 会让入口卡语义保留下来，容易再次混淆，因此放弃。

方案三：新增轻量 `good-section-summary` 收尾块，并加强 tabs 后空隙检查。

优点是既完成说明迁移，又不会再承担入口功能；检查范围更贴近用户截图中的视觉问题。最终选择方案三。

## 5. 最终决策

每个 `good-category-panel` 末尾新增一个 `good-section-summary`：

- `good-furniture`：家具说明
- `good-flooring`：地板说明
- `good-whole-decoration`：整装说明
- `good-outdoor`：户外说明
- `good-collection`：收藏说明
- `good-cultural`：文创说明

顶部结构继续只保留标题、短说明和六个小栏目按钮，按钮后直接进入第一个栏目 panel。

## 6. 具体实现

修改 `solutions/index.html`：在六个栏目 panel 的 `相关参考` 后分别追加轻量说明块，不使用 `good-category-card` 和 `good-category-more`，也不添加“更多内容”按钮。

修改 `styles.css`：新增 `good-section-summary` 的轻量收尾样式，与 `good-section-more`、`good-section-reference` 保持间距关系，但视觉上不再像入口大卡。

修改 `custom/check-content-depth.mjs` 和 `custom/check-release-readiness.mjs`：检查范围改为 `good-category-tabs` 结束后到第一个 `good-category-panel` 之前，并新增六个标题残留、六段说明文案、`good-category-card`、`good-category-more` 和“更多内容”的拦截。

## 7. 本轮优点

顶部更确定：六类按钮后不允许出现任何大卡残留。

说明内容没有丢失，而是回到对应栏目内部末尾，符合“入口归入口，内容归内容”的结构。

检查脚本更贴近视觉问题，能防止按钮下方再次插入卡片、标题或旧说明文案。

## 8. 本轮缺点与代价

每个栏目内部新增一个说明块，栏目内容长度略有增加。这是为了保留被移走卡片的说明语义，且只在当前 active 栏目内显示，整体页面不会回到六类全部铺开的状态。

旧的 `good-category-card` CSS 仍保留在样式表中，没有做无关清理，以避免影响历史样式或后续未知复用。

## 9. 验证与结果

本轮需要运行完整本地验证：

- `node --check script.js`
- `node --check data\site-content.js`
- `node --check forms\form.js`
- `node custom\check-home-links.mjs`
- `node custom\check-site-links.mjs`
- `node custom\check-encoding.mjs`
- `node custom\check-content-depth.mjs`
- `node custom\check-release-readiness.mjs`
- `node custom\check-navigation-consistency.mjs`
- `node custom\check-clickable-topic-items.mjs`

人工复核重点：顶部六个按钮后直接进入第一个 `good-category-panel`；六个 `good-section-summary` 均在对应 panel 内部末尾；`good-cultural` 仍为唯一文创公开锚点。

## 10. 本轮结论

顶部分类入口不能承载说明卡。分类说明应以轻量内容块留在对应栏目内部，且检查脚本必须围绕真实视觉空隙做约束。

## 11. 下一轮建议

1. 推送后等待 GitHub Pages 部署完成，再强制刷新公网 `solutions/index.html` 验证是否仍有旧缓存。
2. 若公网仍出现六张大卡，优先检查 GitHub Pages 部署时间、浏览器缓存和实际访问分支，而不是继续改业务 HTML。
3. 后续若要优化栏目内部密度，只调整当前 active panel 内部，不再新增顶部入口卡。
