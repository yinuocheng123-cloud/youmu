# 版本记录：v1.15.4-public-launch-frontstage-polish

## 1. 本轮目标

本轮目标是完成“柚喜饰界｜V1.15.4 正式上线前前台口径与页面精修版”。重点不是扩展内容、重做结构或新增功能，而是把已经搭好的公开前台从“后期预览站口气”继续推向“可以推给用户或企业查看的公开预览版”。

本轮优先级最高的问题，是全站公开页面仍残留一些建设期语言、后台说明语言、资料审核语言和未完成口径，例如资料框架、待确认、授权后上线、正式联系方式待补、会员站申请页等。这些表达会让用户觉得网站仍在搭建中，不像正式项目页面。

本轮属于前台表达精修和发布前质量收口，不新增后台、登录、商城、支付、数据库，不改 GitHub Actions workflow，不改 sitemap，不改导航结构，不改 `forms/form.js` 核心摘要逻辑，也不新增真实厂商、真实案例、真实认证或真实资质。

## 2. 背景与上下文

本轮承接 `9e74a04 V1.15.3G 标题单行与版式对齐收口`。V1.15.3G 已经处理了桌面端短标题、CTA 底线、模块头部和移动端断行等视觉细节；再往前的 V1.15.3F-3 已经把重点知识文章推向自然长文体。

因此，本轮不再继续解决“页面是否能打开”“导航是否一致”“标题是否孤行”这类结构或版式问题，而是转向前台话术的正式化：页面需要少解释网站后台逻辑，多呈现用户能直接理解的内容价值、企业资料方向和社群承接路径。

## 3. 问题分析

表面问题是若干词语不适合公开前台，例如“资料框架”“待确认”“正式联系方式将在人工确认后补充”。更深层的问题是，一些页面仍在解释网站如何组织资料、如何等待授权、如何做人工复核，而不是直接让用户阅读内容、理解方向和进入下一步。

推荐厂商页如果先讲资料体系，会像后台规划页；公开资料观察库如果反复写确认状态，会像审核表；知识和专题页如果出现“阅读路径”“沟通清单”“人工验收”，会把文章感打散；厂商申请页如果过多强调自动流程，又容易被误解为自动入驻或平台认证。

本轮的核心矛盾是：既要保留必要风险边界，又不能让风险边界散落到每张卡片和正文中，破坏正式前台观感。因此本轮把风险边界尽量压缩到底部一句话，把主内容改成更自然的前台表达。

## 4. 候选方案比较

方案一是只补强 `check-release-readiness.mjs`，让脚本拦截旧词。这种方案成本最低，但会变成“检查通过了，页面仍像说明书”，不能解决真实阅读体验，因此放弃。

方案二是重写所有知识、专题、厂商、观察库和案例页面。它能最彻底地去掉旧口吻，但会大幅扩大改动范围，容易破坏 V1.15.3F 和 V1.15.3G 已经稳定下来的结构，也超出“上线前精修”的边界，因此放弃。

方案三是“全站搜索定位 + 重复段落批量收口 + 关键页面人工复查 + 检查脚本补强”。它能把同类旧表达一次性清掉，又不改变栏目路径和页面结构，是本轮最终采用的方案。

## 5. 最终决策

本轮采用方案三。公开 HTML 与 `data/site-content.js` 中的旧口径统一清理；推荐厂商页、公开资料观察库、厂商申请页、文章和案例页保留现有结构，只改表达方式；检查脚本补充 V1.15.4 的前台禁用词，避免后续回退。

对于“平台认证”“交易担保”等风险词，本轮保留简洁否定语境，例如“不代表平台认证”“不构成交易担保”，但不让它们在卡片中反复出现。对于厂商申请页，“生成摘要”是既有静态表单功能，继续保留；`forms/form.js` 最终恢复为不修改状态。

## 6. 具体实现

- `index.html`：清理首页认识柚喜、推荐厂商、Footer 和社群交流区中的建设期口吻，推荐厂商模块改成按方向查看企业资料。
- `about/index.html`：将品牌介绍页中的资料边界、会员站英文口径等表达改成正式品牌说明。
- `knowledge/` 与 `knowledge/topics/`：清理知识聚合页和细分文章中的阅读路径、人工验收、沟通清单、资料底稿等说明书式表达，改为文章或知识判断口吻。
- `solutions/` 与 `solutions/guides/`：将专题页和指南页中的阅读路径、后台、样板字段等表达改成场景参考和产品判断表达。
- `vendors/index.html`：改成“推荐厂商”正式前台页，按五类柚木方向查看企业资料，并将边界说明压缩到底部。
- `vendors/candidates/index.html`：改成“公开资料观察库”表达，卡片保留品牌名、观察内容和来源链接，底部统一说明观察边界。
- `vendors/*-sample.html`：去掉会员站样板、真实入驻、授权后上线等口径，改成企业资料展示示例的正式表达。
- `articles/` 与 `cases/`：清理文章列表和案例页中的说明书口吻、授权后上线、展示框架等表达。
- `forms/vendor-apply.html` 与 `forms/consult.html`：申请页继续保留摘要生成流程，表达为提交资料后通过社群交流继续沟通；备用咨询页不再讲后台线索系统。
- `data/site-content.js`：清理面向前台数据源中的旧栏目名、建设期备注和未完成口径。
- `custom/check-release-readiness.mjs`：补充 V1.15.4 前台禁用表达，扫描公开 HTML 与 `data/site-content.js`，不误报 `custom/` 内部记录。
- `custom/client-preview-note.md`、`custom/launch-checklist.md`、`custom/public-review-checklist.md`：追加 V1.15.4 说明和人工复核项。

## 7. 本轮优点

本轮让公开前台更像正式网站，而不是项目搭建说明。用户打开首页、推荐厂商、观察库、文章或申请页时，会先看到内容价值、方向分类和可理解的下一步，而不是资料审核、待确认、授权后上线等内部流程。

改动没有改变 URL、导航、栏目结构或表单核心逻辑，风险相对可控。发布预检也同步补强，后续如果再次写回“资料框架”“待确认”“公开资料候选”“好物方案”等旧表达，会在本地验证阶段被拦截。

## 8. 本轮缺点与代价

本轮涉及的公开页面较多，主要原因是旧口径在知识、方案、文章、案例和厂商样例页中重复出现。虽然每处都是小改，但整体 diff 较大，后续需要通过公网抽查确认没有个别句子因为批量替换而显得生硬。

另一个代价是：为了保持正式前台观感，一些内部真实性说明被压缩到底部简短边界。这样阅读更顺，但运营侧仍需在 `custom/` 内部文档中继续维护资料来源、图片授权、服务范围和企业资料确认清单，不能因为前台简洁就忽略真实资料核查。

## 10. 本轮结论

V1.15.4 的结论是：当前阶段最重要的不是继续增加页面数量，而是让已有公开页面停止解释“网站还在怎样搭建”，转为直接呈现“用户现在可以看什么”。推荐厂商、公开资料观察库、文章、案例和厂商申请页都需要以前台阅读体验为主，内部确认流程只保留为底部简洁边界或 `custom/` 内部文档。

本轮没有新增后台、登录、商城、支付、数据库，没有修改 GitHub Actions workflow、sitemap 或 `forms/form.js` 核心逻辑，也没有新增真实厂商、真实案例、真实认证或真实资质。

资料真实性处理说明：所有企业、品牌、案例和图片相关内容仍需结合公开来源、企业资料、图片授权、联系方式、服务范围、合同和售后说明综合确认。公开资料观察库不代表柚喜饰界推荐，不构成平台认证或交易担保。

## 11. 下一轮建议

1. GitHub Pages 部署后，优先人工抽查首页、推荐厂商、公开资料观察库、厂商申请页、文章页和案例页，确认没有建设期口吻回潮。
2. 如果继续精修，建议做一次 `styles.css` 历史覆盖层整理，把 V1.15.2 到 V1.15.3G 的视觉规则合并成更清晰的设计系统层。
3. 企业资料真实上线前，应优先补齐企业资料来源、图片授权、联系方式、案例材料和服务范围说明，而不是继续增加候选品牌数量。
