# 版本记录：v1.15.3f-3-knowledge-article-natural-longform

## 1. 本轮目标

本轮目标是完成“V1.15.3F-3 知识文章自然长文体收口”。前两轮已经把保养主题和产地主题从碎片页合并到主文档，但公网页面仍能看出目录、锚点、小节标题和“继续看”式跳转带来的拼装感。本轮要进一步把重点知识文章调整为更自然的连续文章体。

本轮属于前台内容体验修复和可维护性提升，不新增后台、登录、商城、支付、数据库，不修改 GitHub Actions workflow，不修改 sitemap，不修改 `forms/form.js`，也不新增真实厂商、真实案例、真实认证或真实资质。

## 2. 背景与上下文

本轮承接 `85918bd V1.15.3F-2 产地基础文章长文档化收口`。V1.15.3F 已经形成“一个主题，一篇主文章”的方向；V1.15.3F-1 处理了保养文章；V1.15.3F-2 处理了产地文章。但这些文章仍存在一个新的体验问题：虽然内部核验话术减少了，页面结构却仍像知识点清单，用户阅读时会被多个小节标题、目录锚点和跳转链接打断。

因此，本轮不是继续扩充文章数量，而是把既有核心文章向“可连续读完”的文章体推进，并让首页、知识首页和知识聚合页不再把小问题强制跳到文章内部锚点。

## 3. 问题分析

表面问题是 `teak-origin-basic.html` 和 `teak-daily-cleaning.html` 仍然有小节化痕迹；更深层的问题是知识文章的组织方式容易把一个完整主题拆成很多小问题，导致用户读到的是“知识点拼装”，不是文章。

如果继续保留大量 `<h2>`、文章目录和 `html#anchor` 链接，短期看起来更好定位，长期会让每篇文章都像说明书页，也会诱导后续迭代继续为每个小问题制造独立小页或锚点。本轮要改掉这个默认路径。

## 4. 候选方案比较

方案一是只清理发布预检中会报错的词，不调整结构。优点是改动最小，缺点是无法解决文章仍然小节化的问题，因此放弃。

方案二是删除旧辅助页并重建文章体系。优点是结构最干净，缺点是会带来 sitemap、旧链接和 404 风险，也超出本轮“不新增/不删除公开页面”的边界，因此放弃。

方案三是在现有 URL 内调整文章体和链接关系：主文章自然长文化，旧辅助页继续保留，首页和知识入口改为直接指向主文章，不再指向内部锚点。这个方案风险低，能解决当前可见问题，最终采用。

## 5. 最终决策

本轮采用“主文章自然长文体 + 旧辅助页保留 + 检查脚本兜底”的方案。`teak-origin-basic.html`、`teak-daily-cleaning.html` 作为最重点页面去掉文章内目录和小节化结构；其他核心文章把明显的小节标题转为正文内弱标题或自然相关主题，减少标题堆叠。

首页和知识聚合页中的保养相关小问题统一指向 `teak-daily-cleaning.html`，不再指向 `#daily-cleaning`、`#color-change`、`#outdoor-care`、`#oil-care` 等内部锚点。这样用户看到多个小问题时，仍会进入同一篇保养主文章，而不是被带入碎片式阅读路径。

## 6. 具体实现

- `knowledge/topics/teak-daily-cleaning.html`：移除文章目录和内部锚点式小节，把日常清洁、颜色变化、户外维护、油养、水渍磨损和保养边界融入连续正文，只保留简洁相关内容区。
- `knowledge/topics/teak-origin-basic.html`：移除目录化小节，把常见产地、天然林/人工林、等级、工艺、不同用途和买前资料判断写成连续正文，只保留简洁相关内容区。
- `knowledge/topics/teak-price-difference.html`、`teak-authenticity-basic.html`、`teak-buying-pitfalls.html`、`what-is-teak.html`、`flooring-fit-space.html`、`brand-or-factory.html`：检查并弱化大量小节标题，把“继续看”调整为“相关主题”。
- `index.html`、`knowledge/index.html`、`knowledge/teak-care.html`、`knowledge/teak-material-craft.html`：将保养相关入口从文章内锚点改为主文章链接。
- `knowledge/` 下若干聚合页和辅助页：清理“继续看”式表达，改成更自然的空间理解或相关主题表达。
- `styles.css`：补充自然长文体段落、正文宽度、行距和轻量相关内容样式。
- `custom/check-content-depth.mjs`：增加自然长文体知识文章检查，限制目录结构和过多 `<h2>` 回潮。
- `custom/check-release-readiness.mjs`：增加自然长文体相关文章的目录式词语拦截。
- `custom/check-clickable-topic-items.mjs`：允许多个保养小问题指向同一篇保养主文章，而不是要求内部锚点。

## 7. 本轮优点

文章阅读体验更接近真实前台文章。用户打开保养和产地页面后，不需要先看目录、锚点和说明，也不会被引导到多个碎片小页中拼完整主题。

链接维护也更稳：多个同主题小问题可以指向同一篇主文章，减少内部锚点失效风险。检查脚本补强后，后续如果重新加入目录结构或过多小节标题，会在本地验证阶段暴露。

## 8. 本轮缺点与代价

本轮没有删除旧辅助页，因此站点中仍保留一些较短的兼容页面。这样做是为了避免旧链接和 sitemap 产生 404，但代价是站点结构不是最极简。

另一个代价是自然长文体检查更严格。未来如果确实需要在某篇文章中使用更多小标题，需要同步评估 `custom/check-content-depth.mjs` 的规则，而不是直接绕过检查。

## 10. 本轮结论

“一个主题，一篇文章”不只是 URL 合并，也包括正文结构合并。保养、产地这类主题更适合用连续正文承载，把小问题自然写进段落，而不是用目录、锚点和大量小节标题拆开。

本轮没有新增后台、登录、商城、支付、数据库，没有修改 workflow、sitemap 或 `forms/form.js`，也没有新增真实厂商、真实案例、真实认证或真实资质。

## 11. 下一轮建议

1. GitHub Pages 部署后，优先抽查 `teak-daily-cleaning.html` 和 `teak-origin-basic.html` 是否已经像完整文章，而不是目录式说明页。
2. 继续抽查其他知识文章中是否还有“相关内容”堆叠过多或弱标题太密的问题。
3. 如果未来要系统收口旧辅助页，应先评估 sitemap、站内链接和外部已分享链接，再决定是否保留、重定向或合并。
