# 版本记录：v1.7.1b-2-site-link-check

## 1. 本轮修改目标

本轮目标是在 `V1.7.1B-1 静态表单摘要生成` 后，单独补齐全站链接检查脚本和 sitemap 校验能力。任务类型属于可维护性提升与发布前检查补强，不改表单功能、不扩写内容页、不调整首页视觉、不修改移动端导航，也不改 GitHub Actions workflow。

## 2. 背景与上下文

当前基础为：`V1.7.1A` 已新增知识页、方案页、厂商页、案例页、文章页、表单页和关于页等内容页骨架；`V1.7.1B-1` 已完成咨询表单与厂商申请表的静态摘要生成、复制和清空能力。

本轮需要在不继续拉长内容建设的前提下，确认公开 HTML 页面之间没有空链接、坏链接、错误路径，同时确认 sitemap 只暴露公开页面。

## 3. 问题分析

V1.7.1A 新增页面较多，单靠首页链接检查已经不足以覆盖全站。当前主要风险包括：某个内容页相对路径写错、表单页或内容页出现空链接、sitemap 误把 `custom/`、截图、过程记录或本地路径暴露到公网。

核心矛盾是：站点仍是纯静态项目，检查能力也应保持轻量，不引入依赖、不联网、不接入后台，只在本地 Node 环境中完成闭环校验。

## 4. 候选方案比较

方案一：继续只使用 `custom/check-home-links.mjs`。优点是现有脚本稳定、检查快；缺点是只能覆盖首页，无法发现内容页之间的路径错误，也无法校验 sitemap 暴露风险。本轮放弃该方案作为唯一检查方式。

方案二：新增轻量全站检查脚本。优点是可以递归覆盖公开目录、检查本地链接和 sitemap，且不引入第三方依赖；缺点是使用正则解析 HTML，不能替代完整浏览器渲染检查。本轮选择该方案，因为它符合纯静态、最小改动和低上下文成本要求。

## 5. 最终决策

新增 `custom/check-site-links.mjs`，沿用项目现有检查脚本风格，使用 Node 内置模块完成公开 HTML 收集、href 提取、本地链接存在性检查和 sitemap URL 校验。`custom/check-home-links.mjs` 保留不删除，并继续作为首页专项检查。

## 6. 具体实现

- `custom/check-site-links.mjs`：新增全站链接与 sitemap 检查脚本。
- `custom/client-preview-note.md`：补充 V1.7.1B-2 已新增全站链接检查脚本、sitemap 校验和纯静态边界说明。
- `custom/launch-checklist.md`：补充发布前检查清单中的全站链接与 sitemap 检查项。
- `custom/notes/v1.7.1b-2-site-link-check.md`：新增本轮版本记录。

脚本检查范围包括 `index.html`、`knowledge/**/*.html`、`solutions/**/*.html`、`vendors/**/*.html`、`cases/**/*.html`、`articles/**/*.html`、`forms/**/*.html`、`about/**/*.html` 和 `sitemap.xml`。

## 7. 本轮优点

本轮让全站链接检查从“首页级”扩展到“公开页面级”，并把 sitemap 暴露风险纳入本地验证。脚本输出包含 HTML 文件数、本地链接数和 sitemap URL 数，便于后续迭代快速确认站点是否仍然闭环。

## 8. 本轮缺点与代价

脚本使用轻量正则提取 `href`，不是完整 HTML 解析器；它适合当前静态站点，但如果后续出现复杂模板、运行时生成链接或特殊 HTML 语法，可能需要升级为 DOM 解析方案。脚本不联网检查外链可用性，也不做真实浏览器点击测试。

## 10. 本轮结论

V1.7.1B-2 已完成全站链接检查脚本版。当前站点仍为纯静态网站，没有新增后台、登录、商城、支付或数据库；没有新增购物车、价格、“立即购买”等电商交易能力。
