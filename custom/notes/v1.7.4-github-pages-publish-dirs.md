# 版本记录：v1.7.4-github-pages-publish-dirs

## 1. 本轮目标

本轮目标是完成“柚喜饰界｜V1.7.4 GitHub Pages 发布目录补齐版”。公网首页已经可以打开，但点击 V1.7 新增栏目后出现 404；本地 `check-site-links` 通过，说明仓库内页面和链接闭环大概率正常，问题重点转向 GitHub Pages 的 `_site` 发布目录。

## 2. 背景与上下文

V1.5 为避免把 `custom/`、记录文件、截图和预览包发布到公网，GitHub Actions workflow 改为白名单复制。V1.7 后新增了 `knowledge/`、`solutions/`、`vendors/`、`cases/`、`articles/`、`forms/`、`about/` 等公开页面目录，但 workflow 仍只复制首页、基础资源、`assets/` 和 `data/`，导致公网发布包缺少这些目录。

本轮承接 V1.7.3 导航结构回正结果，只修发布目录，不改首页内容、导航结构、表单功能和 sitemap。

## 3. 问题分析

本地链接检查通过但公网二级页面 404，通常说明源文件存在，但部署产物缺少对应文件。检查 `.github/workflows/pages.yml` 后确认，`Prepare static site artifact` 步骤只执行了：

- 复制 `index.html`、`styles.css`、`script.js`、`404.html`、`robots.txt`、`sitemap.xml`、`.nojekyll`
- 复制 `assets/`、`data/`

它没有复制 V1.7 新增的公开页面目录，因此 GitHub Pages artifact 中不会包含这些页面。

## 4. 候选方案比较

方案一：把整个仓库复制到 `_site`。

优点是简单，不会漏目录。缺点是会重新暴露 `custom/`、`.github/`、记录文件、截图、预览包等不应发布的内容，与 V1.5 的收口原则冲突。因此放弃。

方案二：继续使用白名单，只补齐 V1.7 公开目录。

优点是保持发布边界清楚，只发布公网运行需要的静态文件；缺点是后续再新增公开目录时仍需要同步维护白名单。本轮采用该方案。

## 5. 最终决策

继续沿用白名单发布策略，在 `.github/workflows/pages.yml` 中为 `_site` 补充复制：

- `knowledge/`
- `solutions/`
- `vendors/`
- `cases/`
- `articles/`
- `forms/`
- `about/`

`custom/`、`.git/`、`.github/`、截图、过程记录、甲方预览包和本地说明文档继续不进入发布目录。

## 6. 具体实现

修改 `.github/workflows/pages.yml` 的 `Prepare static site artifact` 步骤，在原有：

```bash
cp -R assets data _site/
```

后新增：

```bash
cp -R knowledge solutions vendors cases articles forms about _site/
```

同时更新 `custom/client-preview-note.md` 和 `custom/launch-checklist.md`，补充 V1.7.4 发布目录修复说明。

## 7. 本轮优点

本轮改动极小，只修复部署产物白名单，不触碰页面内容、导航结构、表单逻辑和 sitemap。公网部署后，V1.7 新增的公开页面目录应随首页一起进入 GitHub Pages artifact，解决公网点击栏目 404 的主要原因。

## 8. 本轮缺点与代价

当前 workflow 仍是手动白名单策略。它的优点是发布边界可控，但后续如果新增新的公开页面目录，仍需要同步更新 workflow，否则可能再次出现本地通过、公网缺文件的问题。

本轮没有实际等待 GitHub Actions 完成并联网复核公网结果；最终公网是否恢复，还需要推送后查看 Actions 部署结果和 GitHub Pages 页面。

## 9. 验证与结果

本轮已验证：

- `node --check script.js`
- `node --check data\site-content.js`
- `node --check forms\form.js`
- `node custom\check-home-links.mjs`
- `node custom\check-site-links.mjs`
- `git diff --check`
- 静态检查 workflow 是否包含 V1.7 新增公开目录，且未复制 `custom/`、`.git/`、`.github/`

验证结果：

- `script.js`、`data/site-content.js`、`forms/form.js` 语法检查通过。
- 首页链接检查通过：共检查 64 个 href，27 个页面 id。
- 全站链接检查通过：检查 32 个 HTML 文件，592 个本地链接，sitemap 32 个 URL。
- `git diff --check` 通过，仅出现 Windows 换行提示。
- workflow 发布白名单检查通过：`knowledge`、`solutions`、`vendors`、`cases`、`articles`、`forms`、`about` 均已包含；`custom`、`.git`、`.github` 未进入复制列表。

## 10. 本轮结论

本轮确认公网 404 的高概率原因是 GitHub Pages `_site` artifact 缺少 V1.7 新增公开目录。通过补齐 workflow 白名单，可以在不扩大发布范围的前提下修复公网页面缺失。

本轮没有新增后台、登录、商城、支付、数据库；没有新增购物车、价格、“立即购买”；没有改首页内容、导航结构、表单功能或 sitemap。

## 11. 下一轮建议

1. 推送后查看 GitHub Actions 的 Pages 部署是否成功。
2. 公网部署完成后抽查 `knowledge/index.html`、`solutions/index.html`、`forms/consult.html` 等二级页面。
3. 后续新增公开目录时，同步更新 `_site` 发布白名单，或考虑维护一个明确的发布目录清单。

## 12. 资料真实性处理说明

本轮只修 GitHub Pages 发布目录，不新增资料内容，不联网核实厂商、案例、认证、资质或图片来源。真实厂商、真实案例、真实认证、最终二维码和联系方式仍需人工确认。
