# 版本记录：v1.7.5-encoding-garbled-text-fix

## 1. 本轮目标

本轮目标是完成“柚喜饰界｜V1.7.5 全站中文乱码修复与编码统一版”。公网二级页面已经能打开，但进入后出现大量 `????????`，需要只围绕公开页面中文编码、乱码源文件和检查脚本进行修复。

本轮不扩写新栏目，不改主导航结构，不改移动端导航逻辑，不改表单摘要逻辑，不改 GitHub Actions workflow。

## 2. 背景与上下文

V1.7.4 已修复 GitHub Pages `_site` 发布目录缺少 V1.7 公开页面目录的问题。页面能打开后，继续暴露出二级页中文内容不可读的问题。

本轮承接 V1.7.3 的导航结构和 V1.7.4 的发布目录，只处理编码与乱码。

## 3. 问题分析

先执行 `git grep -n "????????" -- "*.html" "*.js" "*.css"` 后，确认大量公开 HTML 源文件中已经存在真实连续问号。因此这不是单纯浏览器缺少 `<meta charset="UTF-8">` 的显示问题，而是部分源文件内容已经被错误写成问号。

同时在 PowerShell 终端中查看 UTF-8 中文时会出现 mojibake 显示，但用 Node 以 UTF-8 读取并检查关键中文文本时可以确认文件本体是否正常。因此本轮判断以严格 UTF-8 解码和源文件字符检查为准。

## 4. 候选方案比较

方案一：只补 `<meta charset="UTF-8">`。

优点是改动小。缺点是无法修复源文件已经存在的真实 `????????`，公网仍会显示问号。因此放弃。

方案二：直接通过 PowerShell here-string 重新写中文。

优点是可以一次性生成页面。缺点是当前终端编码会把中文再次替换为 `?`，本轮试验已验证该路径不可靠。因此放弃。

方案三：从 Git 历史对象读取正常 UTF-8 原始字节恢复，再保留 V1.7.3 导航和 V1.7.1B 表单功能。

优点是绕开终端编码转换，能够恢复真实 UTF-8 中文；同时不回滚表单功能和主导航结构。本轮采用该方案。

## 5. 最终决策

本轮从历史提交中恢复正常 UTF-8 内容：

- 内容骨架主要参考 `126796c V1.7.1A 内容页骨架生成`。
- 表单页参考 `f30ee4d V1.7.3 导航结构回正与内容入口降级`，保留静态摘要表单结构。
- 内容页主导航继续使用 V1.7.3 的 5 项结构。

## 6. 具体实现

- 恢复 `knowledge/`、`solutions/`、`vendors/`、`cases/`、`articles/`、`about/` 下公开 HTML 的可读中文内容。
- 恢复 `forms/consult.html` 和 `forms/vendor-apply.html` 的静态表单中文内容与表单字段。
- 统一内容页 Header 导航为“认识柚喜、柚木知识、好物方案、推荐厂商、社群交流”。
- 新增 `custom/check-encoding.mjs`，用于检查公开 HTML、CSS、JS 是否可按 UTF-8 严格读取，HTML 是否包含 `<meta charset="UTF-8">`，以及是否存在连续问号或明显乱码字符。
- 更新 `custom/client-preview-note.md` 和 `custom/launch-checklist.md`，补充 V1.7.5 编码修复说明。

## 7. 本轮优点

本轮修复目标明确，只恢复可读中文和编码检查能力，没有扩张内容体系，也没有改动首页视觉、导航结构、表单逻辑或 GitHub Actions 发布目录。

新增编码检查脚本后，后续可以在本地直接发现连续问号、替换字符和缺失 UTF-8 meta 的问题，避免再次推送到公网后才发现。

## 8. 本轮缺点与代价

由于部分深填内容在历史提交中已经损坏，本轮优先恢复可读中文骨架，而不是完整复原 V1.7.2 的所有深填文案。代价是部分页面内容深度回到较基础状态。

这符合本轮“修复可读中文，不重新做一版内容站”的目标；后续若要恢复深度内容，应单独做内容页文案复核版。

本轮已运行：

- `node --check script.js`

- `node --check forms\form.js`
- `node custom\check-home-links.mjs`
- `node custom\check-site-links.mjs`
- `node custom\check-encoding.mjs`

验证结果：

- 首页链接检查通过：共检查 64 个 href，27 个页面 id。
- 全站链接检查通过：检查 32 个 HTML 文件，558 个本地链接，sitemap 32 个 URL。
- 编码检查通过：公开页面均为 UTF-8，未发现连续问号或明显乱码。

## 10. 本轮结论

本轮问题类型判断为：部分公开页面源文件已经存在真实连续问号，不是单纯浏览器解码问题。修复方式是从历史 Git 对象恢复正常 UTF-8 原始内容，并新增编码检查脚本。

本轮没有新增后台、登录、商城、支付、数据库；没有新增购物车、价格、“立即购买”；没有新增真实厂商、真实案例、真实认证或真实资质。

## 11. 下一轮建议

1. 推送后等待 GitHub Pages 部署完成，再用手机端和电脑端抽查主要二级页面。
2. 后续如需恢复 V1.7.2 的深填质量，应单独做“内容页文案口径统一与深度恢复版”。
3. 后续每次发布前运行 `node custom\check-encoding.mjs`。
