# 版本记录：V1.16.1 真实推荐厂商补充与上线可信度收口

## 1. 本轮目标

本轮目标是在 V1.16 正式上线冲刺版基础上，补充六家推荐厂商资料页，并进一步收口上线可信度。核心不是扩大宣传，而是把“推荐厂商”做成更可靠的资料入口：能核到的才写，核不到的保持保守表达。

## 2. 背景与上下文

本轮承接 `ab1f518 V1.16 正式上线冲刺版`。V1.16 已新增企业合作页和四个厂商资料页，但厂商资料仍较克制，且公网反馈 `cooperation/index.html` 链接打不开。V1.16.1 继续补齐佤臣柚木、上海庄信柚木，并统一六家厂商页的来源说明与资料边界。

## 3. 问题分析

推荐厂商页一旦写入未经来源支撑的历史、规模、案例、联系方式或认证，短期看会显得丰满，长期会损害可信度。本轮公开检索未找到稳定可引用的企业官网、公众号公开页或公开品牌介绍，因此不能把企业线索写成确定事实。

公网合作页打不开的问题，从当前仓库看 `cooperation/index.html` 已存在且 sitemap 已包含，公开链接失败更像 GitHub Pages 尚未呈现最新提交或部署源未更新。代码侧继续保证页面、链接和 sitemap 完整。

## 4. 候选方案比较

方案一是把用户提供的线索直接写成企业事实。优点是页面更丰富，缺点是存在编造风险，因此放弃。

方案二是完全不新增佤臣和上海庄信页面。优点是最安全，缺点是不满足六家推荐厂商均有独立页面的目标，因此放弃。

方案三是新增页面，但只采用公开资料整理和资料线索口径，明确来源缺口，并通过检查脚本防止强事实回流。该方案符合“宁可少写，不可编造”的原则，因此采用。

## 5. 最终决策

本轮新增 `vendors/wachen-teak.html` 和 `vendors/shanghai-zhuangxin-teak.html`，并重写原四家厂商页主体结构。六家页面统一包含“品牌方向、适合关注、公开资料看点、仍需补充、来源说明”。没有稳定来源的地方不写具体历史、案例、认证、联系方式或规模。

## 6. 具体实现

- `vendors/index.html`：升级为六家推荐厂商资料入口。
- `index.html`：首页推荐厂商模块露出佤臣柚木、上海庄信柚木、壹信柚木、臻藏柚木生活。
- `vendors/wachen-teak.html`：新增佤臣柚木资料页。
- `vendors/shanghai-zhuangxin-teak.html`：新增上海庄信柚木资料页。
- `vendors/zhenzang-teak-life.html`、`vendors/yuebaijia-teak-flooring.html`、`vendors/xuelianhua-teak-furniture.html`、`vendors/yixin-teak.html`：统一资料结构和来源说明。
- `cooperation/index.html`：补充欢迎企业提供公开资料或授权资料的正式表达。
- `vendors/candidates/index.html`：明确观察库是辅助观察页，不等同于推荐厂商。
- `sitemap.xml`：加入两个新增厂商页。
- `custom/vendor-real-source-log.md`：新增六家厂商来源记录。
- `custom/check-release-readiness.mjs`、`custom/check-content-depth.mjs`、`custom/check-site-links.mjs`：补强厂商页存在性、来源说明、sitemap 和强事实风险检查。

## 7. 本轮优点

六家厂商都有独立页面，推荐厂商首页更像品牌馆入口。页面保留用户可理解的关注方向，同时避免把资料不足的品牌写成已经正式合作或已经被平台背书。

检查脚本增加了针对厂商页的来源说明和强事实风险拦截，后续如果误写“已认证、官方推荐、平台背书、工厂规模、认证资质”等内容，会在本地预检阶段暴露。

## 8. 本轮缺点与代价

由于未找到稳定公开来源，六家厂商页仍然偏保守，页面内容不如真实企业授权资料丰富。这个代价是有意接受的，因为当前阶段可信度比丰满度更重要。

公网合作页打不开的问题无法仅通过本地代码完全判断，仍需要在 GitHub Pages 部署完成后人工复查。如果 Pages 源分支不是 `master` 或部署延迟，需要在仓库设置或 Pages 部署记录中进一步确认。

## 9. 验证与结果

本轮计划运行语法、全站链接、编码、内容深度、发布预检、导航一致性和知识条目可点击检查。验证结果以最终命令输出为准。

公网预览地址：https://yinuocheng123-cloud.github.io/youmu/

## 10. 本轮结论

V1.16.1 的核心结论是：推荐厂商可以先建立资料页，但不能为了“看起来真实”而写入未核实事实。公开来源不足时，页面要清楚表达资料边界，把后续补充项留给企业公开资料或授权资料。

本轮没有新增后台、登录、商城、支付、数据库，没有修改 GitHub Actions workflow，也没有修改 `forms/form.js` 核心摘要逻辑。

## 11. 下一轮建议

1. 企业提供官网、公众号或授权资料后，逐页补充可引用来源链接和已核实事实。
2. GitHub Pages 部署完成后，优先打开 `cooperation/index.html`、六家厂商页和 `vendors/index.html` 做公网复查。
3. 如合作页仍打不开，检查 GitHub Pages 发布源、部署分支和缓存状态。
